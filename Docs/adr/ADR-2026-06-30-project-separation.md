# ADR-2026-06-30: PlataVM ведётся как отдельный проект

Статус: принято

Дата: 2026-06-30

## Контекст

При работе через ИИ-ассистента возник риск смешения двух разных проектов:

1. `BuoyCalc Windows` — расчётное приложение для морской постановки.
2. `PlataVM` — аппаратный проект платы питания.

Эти проекты имеют разную природу:

- BuoyCalc Windows относится к расчётной модели, solver, UI, 2D/PDF-отчётам;
- PlataVM относится к схемотехнике питания, аккумуляторам, DC/DC, защитам, интерфейсам и испытаниям.

Их смешение опасно, потому что приводит к потере инженерной трассируемости и ошибочному переносу архитектурных решений.

## Решение

Зафиксировать:

```text
PlataVM ≠ BuoyCalc Windows
```

PlataVM ведётся в отдельном репозитории:

```text
Gonoboben/plataVM
```

Для PlataVM используются отдельные документы:

1. `README.md`;
2. `Docs/PROJECT_CONTROL.md`;
3. `Docs/ARCHITECTURE_BASELINE.md`;
4. `Docs/CHRONOLOGY.md`;
5. `Docs/adr/`;
6. `.github/PULL_REQUEST_TEMPLATE.md`.

## Последствия

Положительные:

1. У проекта платы питания появляется собственная архитектурная база.
2. Хронология PlataVM не загрязняется задачами BuoyCalc.
3. Любое изменение питания, защит, интерфейсов и аккумуляторов будет отслеживаться отдельно.
4. Снижается риск случайного архитектурного дрейфа.

Ограничения:

1. Задачи BuoyCalc не должны выполняться в этом репозитории.
2. Задачи PlataVM не должны выполняться в репозитории BuoyCalc Windows.
3. Если проектная задача затрагивает оба проекта, это должно быть оформлено как связанная задача в двух репозиториях, но не как единая общая архитектура.

## Проверка

При каждом новом шаге по PlataVM проверять:

1. активный проект — `PlataVM / плата питания`;
2. активный репозиторий — `Gonoboben/plataVM`;
3. задача относится к аппаратной плате питания;
4. не затрагивается solver/UI/PDF/2D BuoyCalc;
5. хронология обновляется только в PlataVM.
