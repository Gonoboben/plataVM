# Архитектурная база ПДУ БНПА / PlataVM

Дата фиксации: 2026-06-30

Статус:

```text
V1.4 chat-consolidated baseline
```

Документ фиксирует базовую архитектуру проекта платы питания БНПА.

## 1. Граница архитектуры

Репозиторий:

```text
Gonoboben/plataVM
```

Проект:

```text
ПДУ БНПА / PlataVM / плата питания / энергетический модуль БНПА
```

## 2. Назначение

Плата предназначена для разработки системы распределения питания в составе БНПА.

Основные функции:

1. приём питания от двух основных аккумуляторных блоков;
2. управляемое параллельное подключение BAT1 и BAT2;
3. формирование основной шины BAT_BUS_4S;
4. распределение питания по силовым и вспомогательным нагрузкам;
5. формирование 32 В для ИЗС;
6. резервное питание критических узлов от EMG;
7. управление силовыми каналами;
8. защита силовых цепей;
9. телеметрия состояния питания;
10. связь с главным управляющим модулем;
11. безопасное отключение при авариях.

## 4. Каноническая структура питания

Основная силовая подсистема:

```text
BAT1 + BAT2 → управляемое параллельное подключение → BAT_BUS_4S
```

Критическая подсистема:

```text
EMG_4S2P → CRITICAL_INPUT → 5V_CRIT → 3V3_CRIT
```

Силовые нагрузки:

```text
BAT_BUS_4S → BOOST_A 12→32 В → ИЗС 1..3
BAT_BUS_4S → BOOST_B 12→32 В → ИЗС 4..6
BAT_BUS_4S → AUX1..AUX4
```

Ограничение:

```text
EMG_4S2P не питает ИЗС, BOOST 32 В и силовые AUX-нагрузки.
```

## 5. BAT1/BAT2

BAT1 и BAT2 работают как управляемая параллельная батарейная подсистема.

Минимальные элементы каждого плеча:

1. входной предохранитель;
2. TVS-защита;
3. защита от переполюсовки;
4. измерение напряжения;
5. измерение тока;
6. precharge-цепь;
7. основной back-to-back MOSFET-ключ;
8. диагностика состояния BMS;
9. телеметрия в MCU.

Правило включения:

1. проверить напряжение BAT1;
2. проверить напряжение BAT2;
3. проверить разность напряжений;
4. выполнить precharge;
5. проверить отсутствие аварий;
6. включить силовые ключи;
7. контролировать токи после включения.

Рабочий предварительный допуск:

```text
ΔU BAT1/BAT2 перед параллельным включением: 0,2–0,5 В
```

Окончательный допуск должен быть подтверждён расчётом, BMS-документацией и стендовыми испытаниями.

## 6. EMG_4S2P

EMG — отдельный аварийный источник.

Его назначение:

1. питание MCU;
2. питание критической связи;
3. питание критических датчиков;
4. фиксация аварии;
5. передача аварийной телеметрии;
6. безопасное завершение работы.

EMG имеет две разные функции:

1. отдавать энергию в критическую подсистему;
2. заряжаться от BAT_BUS_4S через отдельный управляемый контур.

Для EMG должны быть предусмотрены:

1. отдельный силовой ключ питания;
2. отдельный ключ или контроллер заряда;
3. ограничение зарядного тока;
4. защита от обратного тока;
5. контроль напряжения;
6. контроль температуры;
7. диагностика готовности.

## 7. Критическое питание

Критическое питание должно обеспечивать:

1. 5V_CRIT;
2. 3V3_CRIT;
3. питание MCU;
4. питание интерфейса связи;
5. питание датчиков, необходимых для диагностики;
6. сохранение fault latch;
7. возможность передачи телеметрии после аварии основной шины.

Критическая подсистема не должна зависеть от работоспособности каналов ИЗС.

## 8. ИЗС

ИЗС — основная мощная регулируемая нагрузка.

Архитектура:

1. 6 каналов;
2. каждый канал номинально 30 Вт;
3. питание через 32 В;
4. два boost-преобразователя 12→32 В;
5. группа A: ИЗС 1..3;
6. группа B: ИЗС 4..6;
7. отдельное управление каждым каналом;
8. PWM/диммирование от 5%;
9. контроль тока каждого канала как целевая функция.

Ожидаемые защиты:

1. короткое замыкание канала;
2. перегрузка канала;
3. обрыв канала;
4. перегрев ключа;
5. перегрузка boost-преобразователя;
6. отключение группы при аварии boost.

## 9. AUX

AUX-каналы предназначены для вспомогательных нагрузок.

Предварительная структура:

```text
BAT_BUS_4S → AUX1
BAT_BUS_4S → AUX2
BAT_BUS_4S → AUX3
BAT_BUS_4S → AUX4
```

Для каждого AUX должны быть зафиксированы:

1. назначение;
2. напряжение;
3. номинальный ток;
4. пиковый ток;
5. защита;
6. допустимость отключения;
7. поведение при аварии;
8. необходимость телеметрии.

## 10. Интерфейсы

Основной интерфейс связи:

```text
RS-485
```

Сервисные интерфейсы:

```text
UART / SWD
```

CAN допускается только как опциональное направление после отдельного решения.

Для RS-485 должны быть определены:

1. топология;
2. терминаторы;
3. fail-safe biasing;
4. защита от ESD/Surge;
5. необходимость гальванической развязки;
6. протокол обмена;
7. набор телеметрических регистров;
8. поведение при потере связи.

## 11. Земля и помехозащита

Базовое правило:

```text
Power GND и Signal GND не объединяются произвольно.
```

Для платы должны быть зафиксированы:

1. силовые зоны;
2. сигнальная зона MCU;
3. зона boost-преобразователей;
4. зона ИЗС;
5. зона входных аккумуляторных токов;
6. зона RS-485;
7. точка или схема соединения GND_PWR и GND_SIG;
8. путь возвратных токов;
9. правила трассировки измерительных линий;
10. локальная развязка питания.

## 12. Архитектурные ограничения

Без отдельного архитектурного решения нельзя:

1. подключать EMG к ИЗС;
2. питать BOOST 32 В от EMG;
3. подключать силовые AUX к EMG;
4. заменять BAT_BUS_4S на другую базовую шину;
5. убирать precharge;
6. включать BAT1/BAT2 параллельно без контроля напряжений;
7. объединять земли без схемы и проверки;
8. менять RS-485 как базовый интерфейс;
9. переносить защитную логику MCU в недокументированную внешнюю схему;
10. переходить к PCB без закрытия открытых вопросов, влияющих на силовую схему.

## 13. Следующий инженерный рубеж

Следующий обязательный рубеж:

```text
schematic freeze V1.4
```

Для достижения рубежа нужно закрыть:

1. EMG: ёмкость, зарядный ток, BMS, защита;
2. BAT1/BAT2: BMS, кабели, допустимые уравнительные токи;
3. ИЗС: токи, драйверы, ключи, диагностика;
4. BOOST: топология, токи, КПД, тепло;
5. AUX: состав и режимы;
6. RS-485: схема защиты и развязки;
7. MCU: выбор семейства, питание, ADC, watchdog;
8. PCB: слойность, медь, зоны, тепловой режим.
