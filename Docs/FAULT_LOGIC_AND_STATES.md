# Состояния, аварийная логика и реакции ПДУ БНПА / PlataVM

Дата фиксации: 2026-06-30

Статус:

```text
state-machine and fault baseline for V1.4 chat-consolidated architecture
```

## 1. Назначение

Документ фиксирует рабочую модель состояний и аварийных реакций платы питания.

Цель — не допустить ситуации, когда силовая схема, прошивка и тесты описывают разные устройства.

## 2. Главный принцип безопасности

ПДУ должна переходить из штатного режима в безопасный режим ступенчато:

```text
предупреждение → снижение нагрузки → отключение силовых каналов → критическое питание → аварийная защёлка
```

EMG используется только для критического питания и сохранения управляемости.

## 3. Состояния верхнего уровня

| Состояние | Назначение | Основные действия |
|---|---|---|
| OFF | питание силовых нагрузок выключено | ключи выключены, критическое питание по условиям |
| SELF_TEST | самопроверка | проверка MCU, ADC, связи, датчиков |
| BAT_CHECK | проверка основных АКБ | измерение BAT1/BAT2, BMS, ΔU |
| PRECHARGE_BAT1 | предзаряд от BAT1 | включение precharge BAT1 |
| PRECHARGE_BAT2 | предзаряд от BAT2 | включение precharge BAT2 |
| MAIN_BUS_ENABLE | включение BAT_BUS_4S | включение силовых ключей |
| RUN_PARALLEL | штатная работа BAT1+BAT2 | питание нагрузок, баланс токов |
| RUN_SINGLE_BAT1 | работа только от BAT1 | BAT2 отключен или недоступен |
| RUN_SINGLE_BAT2 | работа только от BAT2 | BAT1 отключен или недоступен |
| LOAD_SHED_1 | снижение нагрузки 1 | снижение/отключение части ИЗС |
| LOAD_SHED_2 | снижение нагрузки 2 | отключение ИЗС/части AUX |
| CRITICAL_ONLY | только критическое питание | силовые нагрузки выключены |
| EMG_RUN | питание критической части от EMG | связь, MCU, аварийная телеметрия |
| FAULT_LATCH | аварийная защёлка | силовые каналы заблокированы до сброса |

## 4. Условия переходов

### 4.1. OFF → SELF_TEST

Условия:

1. подано питание на критическую логику;
2. MCU стартовал;
3. supervisor не держит reset;
4. нет аппаратного запрета запуска.

### 4.2. SELF_TEST → BAT_CHECK

Условия:

1. ADC работает;
2. связь с датчиками есть;
3. RS-485 готов;
4. внутренние напряжения 5V_CRIT и 3V3_CRIT в норме;
5. нет критической аппаратной аварии.

### 4.3. BAT_CHECK → PRECHARGE

Условия:

1. BAT1 или BAT2 присутствует;
2. напряжение в допустимом диапазоне;
3. BMS выбранного аккумулятора сообщает готовность;
4. нет короткого замыкания BAT_BUS_4S;
5. разность напряжений BAT1/BAT2 в допустимом окне при параллельной работе.

### 4.4. PRECHARGE → MAIN_BUS_ENABLE

Условия:

1. напряжение BAT_BUS_4S поднялось до ожидаемого уровня;
2. ток предзаряда не превысил лимит;
3. время предзаряда не превысило таймаут;
4. нет аварии по температуре;
5. нет аварии по BMS.

### 4.5. MAIN_BUS_ENABLE → RUN

Условия:

1. силовой ключ включился;
2. BAT_BUS_4S стабилен;
3. ток не превышает лимит;
4. критическая шина стабильна;
5. телеметрия фиксирует штатное состояние.

### 4.6. RUN → LOAD_SHED

Условия:

1. снижение напряжения BAT_BUS_4S;
2. перегрев силовой зоны;
3. перегрузка boost;
4. превышение тока ИЗС;
5. предупреждение BMS;
6. команда оператора.

### 4.7. RUN/LOAD_SHED → CRITICAL_ONLY

Условия:

1. авария основной шины;
2. недопустимое напряжение BAT_BUS_4S;
3. критический перегрев;
4. отказ обоих основных аккумуляторных плеч;
5. протечка, если датчик реализован;
6. потеря управляемости силовых ключей.

### 4.8. CRITICAL_ONLY → EMG_RUN

Условия:

1. основная шина недоступна;
2. EMG готов;
3. напряжение EMG в допустимом диапазоне;
4. критический DC/DC может питаться от EMG;
5. силовые нагрузки отключены.

### 4.9. Любое состояние → FAULT_LATCH

Условия:

1. короткое замыкание основной шины;
2. аппаратная авария ключа;
3. неконтролируемый ток;
4. критический перегрев;
5. авария, требующая ручного разбора;
6. противоречивые состояния силовых ключей.

| F003 | BAT1 overcurrent | отключить BAT1, зафиксировать событие |
| F004 | BAT2 overcurrent | отключить BAT2, зафиксировать событие |

| F006 | Precharge timeout | запретить включение шины, FAULT_LATCH при повторе |
| F007 | BAT_BUS undervoltage | LOAD_SHED или CRITICAL_ONLY |
| F008 | BAT_BUS overcurrent | отключить силовые нагрузки, FAULT_LATCH |
| F009 | EMG undervoltage | запрет EMG_RUN, аварийная телеметрия если возможно |
| F010 | EMG charge fault | отключить заряд EMG |
| F011 | 5V_CRIT fault | FAULT_LATCH / reset по supervisor |
| F012 | 3V3_CRIT fault | FAULT_LATCH / reset по supervisor |
| F013 | BOOST_A fault | отключить ИЗС 1..3 |
| F014 | BOOST_B fault | отключить ИЗС 4..6 |
| F015 | LED channel overcurrent | отключить конкретный канал |
| F016 | AUX overcurrent | отключить конкретный AUX |
| F017 | Overtemperature warning | LOAD_SHED_1 |
| F018 | Overtemperature critical | CRITICAL_ONLY или FAULT_LATCH |
| F019 | RS-485 timeout | безопасное сохранение режима, затем policy по таймауту |

| F021 | Watchdog reset | запись события, SELF_TEST |
| F022 | Key state mismatch | FAULT_LATCH |

## 6. Политика сброса аварий

Аварии делятся на три класса.

### 6.1. Soft fault

Может быть сброшена командой после устранения причины.

Примеры:

1. временное превышение температуры;
2. отключение отдельного ИЗС;
3. одиночная ошибка связи.

### 6.2. Hard fault

Требует перехода в безопасное состояние и отдельного разрешения на повторный запуск.

Примеры:

1. precharge timeout;
2. перегрузка BAT_BUS;
3. отказ boost.

### 6.3. Latched fault

Не сбрасывается простой командой оператора.

Примеры:

1. короткое замыкание основной шины;
2. противоречие состояния силовых ключей;
3. критический перегрев;
4. протечка;
5. отказ критического питания.

## 7. Политика снижения нагрузки

Предварительная ступенчатая логика:

```text
RUN_PARALLEL
  → LOAD_SHED_1: снизить яркость ИЗС / отключить часть ИЗС
  → LOAD_SHED_2: отключить все ИЗС и некритические AUX
  → CRITICAL_ONLY: оставить только MCU, связь, датчики
  → EMG_RUN: питаться от EMG при недоступной основной шине
```

Точные пороги должны быть определены после расчёта токов, тепла и аккумуляторных характеристик.

## 8. Запрещённые состояния

Запрещено:

1. включать BAT1_MAIN и BAT2_MAIN без проверки ΔU;
2. включать основной ключ без успешного precharge;
3. питать ИЗС от EMG;
4. питать BOOST 32 В от EMG;
5. оставлять ИЗС включёнными при CRITICAL_ONLY;
6. заряжать EMG без ограничения тока;
7. сбрасывать latched fault без условий безопасности;
8. выполнять автоматический повтор включения после короткого замыкания;
9. включать AUX неизвестного назначения без лимита тока.

## 9. Минимальные тесты логики состояний

1. OFF → SELF_TEST → BAT_CHECK без силовой нагрузки.
2. BAT1-only старт.
3. BAT2-only старт.
4. BAT1+BAT2 параллельный старт через precharge.
5. Отказ BAT1 при работе BAT1+BAT2.
6. Отказ BAT2 при работе BAT1+BAT2.
7. Precharge timeout.
8. Переход RUN → LOAD_SHED_1.
9. Переход LOAD_SHED_1 → LOAD_SHED_2.
10. Переход LOAD_SHED_2 → CRITICAL_ONLY.
11. Переход CRITICAL_ONLY → EMG_RUN.
12. Запрет включения ИЗС в EMG_RUN.
13. Fault latch при коротком замыкании.
14. Watchdog reset.
15. Потеря RS-485.

## 10. Правило изменения логики

Любое изменение конечного автомата должно сопровождаться:

1. изменением этого файла;
2. изменением требований;
3. изменением тест-плана;
4. обновлением прошивки;
5. записью в хронологии;
6. отдельным ADR, если меняется принцип безопасности.
