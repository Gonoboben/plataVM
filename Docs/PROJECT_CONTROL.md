# Управление проектом PlataVM

Дата фиксации: 2026-06-30

Этот документ задаёт правила ведения **только проекта PlataVM / плата питания**.

## 1. Контекстный замок

Активный проект:

```text
PlataVM / плата питания
```

Запрещено смешивать этот проект с:

1. BuoyCalc Windows;
2. BuoyCalc iOS;
3. расчётным приложением морской постановки;
4. PDF/2D/solver-логикой BuoyCalc.

Если задача относится к BuoyCalc, работа должна быть остановлена в этом репозитории и перенесена в соответствующий репозиторий.

## 2. Назначение проекта

PlataVM ведётся как аппаратный инженерный проект платы питания / энергетического модуля БНПА.

Каждое изменение должно отвечать на вопросы:

1. Что изменено?
2. Почему это изменено?
3. Какая часть аппаратной архитектуры затронута?
4. Не нарушены ли прежние зафиксированные решения?
5. Как проверено, что питание, защита и логика управления не деградировали?

## 3. Область проекта

Входит:

1. аккумуляторная архитектура;
2. входные и выходные шины питания;
3. DC/DC-преобразователи;
4. силовые ключи;
5. защиты;
6. аварийное питание;
7. земля и экранирование;
8. интерфейсы управления и телеметрии;
9. соединители;
10. BOM;
11. тест-планы;
12. стендовые и натурные испытания.

Не входит:

1. расчёт буя;
2. solver BuoyCalc;
3. Windows UI;
4. PDF-отчёт BuoyCalc;
5. 2D/3D-визуализация постановки.

## 4. Базовое правило архитектуры

Архитектура платы питания не меняется «по ходу дела».

Любое изменение, которое затрагивает аккумуляторную схему, аварийное питание, силовые линии, земли, интерфейсы, защиты или разъёмы, должно сопровождаться записью в:

- `Docs/ARCHITECTURE_BASELINE.md` — если фиксируется новая базовая архитектура;
- `Docs/CHRONOLOGY.md` — если это шаг реализации;
- `Docs/adr/ADR-YYYY-MM-DD-*.md` — если это архитектурное решение.

## 5. Запреты

Запрещено:

1. Объединять силовое и аварийное питание без отдельного ADR.
2. Подключать аварийный источник к мощным нагрузкам без расчёта и отдельного решения.
3. Объединять Power GND и Signal GND без схемы точки соединения.
4. Менять аккумуляторную шину без расчёта токов, защиты и теплового режима.
5. Менять интерфейс связи без проверки совместимости с главным модулем.
6. Переносить в PlataVM решения из BuoyCalc Windows.
7. Хранить в PlataVM хронологию BuoyCalc.

## 6. Правило работы через GitHub

После первичной инициализации работа ведётся через ветки и pull request.

Минимальный цикл изменения:

1. Создать ветку под задачу.
2. Внести минимальный инженерно связанный набор изменений.
3. Обновить хронологию.
4. Добавить/обновить ADR, если меняется архитектура.
5. Провести проверку.
6. Описать в PR:
   - цель;
   - затронутые узлы;
   - архитектурный риск;
   - электрические/тепловые риски;
   - проверку;
   - что специально не изменялось.

## 7. Роль ИИ-ассистента

Перед предложением или внесением изменений по PlataVM ассистент должен свериться с:

1. `README.md`;
2. `Docs/PROJECT_CONTROL.md`;
3. `Docs/ARCHITECTURE_BASELINE.md`;
4. `Docs/CHRONOLOGY.md`;
5. `Docs/adr/`;
6. открытыми issues/PR;
7. последними коммитами.

Ассистент не должен уводить проект в новую архитектуру без отдельного ADR и явного объяснения последствий.

В начале каждого рабочего ответа по этому проекту ассистент должен держать активный контекст:

```text
Активный проект: PlataVM / плата питания
Репозиторий: Gonoboben/plataVM
```
